NSE/BSE Signal Screener

Project timeline, implementation plan, effort estimate, operating cost, and ownership split

Prepared on: 16 April 2026

Proposed first version: Local end-of-day investment screener using Zerodha Kite Connect, Python, DuckDB/Parquet, Streamlit, and Telegram alerts.

Primary objective: Replicate the existing TradingView PineScript signal logic and run it across NSE/BSE stocks automatically after market close for investment watchlist generation.

1. Executive Summary

The goal is to build a private investment screener that replaces manual TradingView chart checks. The current PineScript indicator will be translated into Python and run across a large NSE/BSE stock universe. The scanner will update after market close, apply configurable filters, produce a shortlist, and send an alert through Telegram initially.

Recommendation: Build version 1 as a local Python application first, running from VSCode on the MacBook Air. Use Zerodha Kite Connect for market data, DuckDB/Parquet for local storage, Streamlit for UI, and Telegram for alerts.

The first production-ready version should focus only on end-of-day daily and weekly signals for investment decisions. Intraday trading, live tick processing, and order placement are explicitly out of scope.

3. Key Assumptions

  • The application is for private/internal use, not a public market data product.
  • Zerodha Kite Connect paid API will be used for instrument master and historical candles.
  • The scanner will run after market close, targeted around 6:00 PM IST.
  • Version 1 will run on a MacBook Air with 8 GB RAM.
  • The initial strategy is the existing TradingView PineScript named "Weekly Buy Sell".
  • All strategy and filter parameters must be configurable through YAML files.
  • Telegram is the default alert channel because it is faster and simpler than WhatsApp for MVP.
  • Intraday trading, tick streaming, and automated order execution are not part of the product plan.

5. Strategy To Be Implemented

The current PineScript logic will be implemented exactly first. Filters and enhancements will be separate from the raw signal engine.

Component Current Logic Configurable?
Structure sensitivity Highest high and lowest low of prior N candles. Yes, default 3.
Breakout rule Buy when close crosses above prior structure high. Sell when close crosses below prior structure low. Yes, default close-based.
FVG rule Bullish FVG when current low is greater than high from two candles ago. Bearish FVG when current high is less than low from two candles ago. Yes, FVG lookback default 5.
Signal confirmation Breakout plus recent FVG. Yes.
Repeated signal prevention A new buy is blocked after a buy until a sell occurs, and vice versa. Yes, default enabled.
Demand/supply zones Demand uses min(current low, prior low). Supply uses max(current high, prior high). Yes.

7. Daily Evening Job

The scanner should run after the market closes and after data is likely available from the provider. Recommended start time is 6:00 PM IST.

Step Action Expected Time
1 Load config and validate schedule. Seconds
2 Check Kite token and fail fast if login is required. Seconds
3 Fetch/update instrument master if needed. 1-3 minutes
4 Fetch latest daily candle updates for the stock universe. 10-70 minutes depending on stock count and rate limits.
5 Run data readiness checks. Seconds to 2 minutes
6 Resample daily candles into weekly candles if scan timeframe is weekly. 1-5 minutes
7 Run strategy and filters. 1-10 minutes depending on universe size.
8 Save results and send Telegram alert. Seconds

8. Expected Performance On MacBook Air 8 GB RAM

The main bottleneck is API fetching, not computation. Calculation can be kept efficient by processing stock data in batches and storing candles locally.

Universe Size Estimated Daily Update Time Estimated Scan Compute Time Estimated Total Daily Runtime
500 stocks 3-7 minutes Under 1 minute 5-10 minutes
1,000 stocks 6-10 minutes Under 1-3 minutes 10-15 minutes
2,500 stocks 15-25 minutes 2-5 minutes 20-35 minutes
5,000 stocks 30-50 minutes 5-10 minutes 40-70 minutes

Important: Initial historical backfill can take longer, potentially 1-3 hours for thousands of stocks. This is expected and should not happen every day. Daily runs must use incremental updates.

13. Risk Register

Risk Impact Mitigation
TradingView signal mismatch High Validate candles, weekly resampling, crossover logic, and adjusted/unadjusted price differences.
Kite API rate limits Medium Backfill once, use incremental updates, batch carefully, avoid unnecessary repeated downloads.
Kite token/session failure Medium Add token check before job and send Telegram alert when login is required.
MacBook sleeps during job Medium Use manual run first or keep device awake. Move to VPS for reliability.
Duplicate NSE/BSE listings Low Prefer NSE and include BSE-only stocks. Add ISIN mapping when available.
Noisy signal list Medium Add volume, price, trend, and signal-age filters.
WhatsApp setup complexity Low for MVP Use Telegram first. Add WhatsApp only after the end-of-day scanner is stable.
